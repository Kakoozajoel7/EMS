Event Management System

Project Report

This system is a web-based event management portal built for university events. It separates responsibilities between students and administrators while using one shared login page. Students can register, browse events, RSVP, view booking history, and update their profiles. Administrators can manage event records, upload event images, view event statistics, and monitor RSVP records.

Technology: PHP, MySQL, HTML, CSS, Bootstrap

Server: WAMP / Apache

Database: event_mgmt_db

Roles: Student, Admin

System Overview

Student Role

  • Create a student account and log in.
  • Browse and filter upcoming events.
  • RSVP for selected events.
  • View active bookings and event history.
  • Edit personal profile information.

Admin Role

  • Log in from the same login page.
  • Create, update, and delete events.
  • Upload event images.
  • View dashboard statistics.
  • Review and cancel RSVP records.

Database Structure

users
Id, Name, Email, Password, Phone, RegNo, Gender, CreatedAt, Role
admin
Id, Name, Email, Password
events
Id, Name, Category, Description, EventDate, StartTime, EndTime, Venue, Organiser, RSVPDeadline, CreatedBy, CreatedAt, Image
rsvp
Id, UserId, EventId, RSVPDate

Relationships: events.CreatedBy references admin.Id; rsvp.UserId references users.Id; rsvp.EventId references events.Id. The RSVP table also prevents duplicate RSVP records for the same user and event.

Challenges and Lessons Learned

One challenge was keeping the student and admin workflows connected while still separating their roles and visual designs. This was handled using role-based session variables, a shared login page, and separate authentication checks for students and administrators.

Another challenge was keeping file names, database column names, and page links consistent across the project. The solution was to align all admin pages with the actual project filenames and use the same database connection file throughout the system.

The project helped reinforce the importance of planning database relationships early, using prepared statements for database access, managing sessions carefully, and maintaining a clear folder structure for deployment.
